# Save Coordinate Validation

Status: completed

## Problem

Persisted locations are checked for valid coordinates when they are loaded, but
new locations are currently encoded, written, and published without the same
validation. An invalid coordinate can therefore reach the current-run map and
places UI and leave an unusable file that is silently discarded after restart.

## Scope

- Reject invalid coordinates at the start of `saveLocationOnDisk(_:)`.
- Keep rejected saves free of file creation, in-memory publication, and saved-
  location notifications.
- Preserve atomic writes, unique filenames, chronological publication,
  main-thread mutation, and load-time validation for valid locations.
- Extend the maintained static baseline and documentation for the save-time
  boundary.
- Do not change persistence formats, reverse geocoding, retention, map
  annotations, or notification content.

## Implementation

1. Guard `location.coordinates` with `CLLocationCoordinate2DIsValid` before
   encoding or selecting a destination file in `Journal/LocationsStorage.swift`.
2. Extend `scripts/check-baseline.py` with a method-scoped, ordering-sensitive
   contract proving validation precedes encoding, file creation, writes, and
   publication.
3. Record the behavior and maintenance boundary in `README.md`, `VISION.md`,
   `SECURITY.md`, and `CHANGES.md`.

## Validation

- Run `python3 -m py_compile scripts/check-baseline.py` without retaining
  generated bytecode.
- Run `make lint`, `make test`, `make build`, and `make check` from the checkout
  and run the canonical check from an external directory.
- Verify isolated mutations that remove or invert validation, move validation
  after a side effect, weaken the method-scoped contract, omit maintenance
  guidance, or leave this plan incomplete are rejected.
- Run `git diff --check` and explicit generated-artifact, secret-pattern, and
  intended-path audits.
- Record that Xcode and simulator execution are unavailable on this Linux host;
  rely on the existing macOS hosted baseline for exact-head project parsing.

## Risks

- Core Location validity semantics remain the single source of truth for
  latitude and longitude acceptance.
- The stacked base PR must remain available and merge before this change.
